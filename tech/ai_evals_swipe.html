Thesis I
The Work Is The Looking

Every source, same worldview.

Look at your data. Define binary failure modes with a domain expert. Align scalable graders to that expert's judgment. The metric dashboard is a downstream artifact.

Generic metrics like ROUGE and BERTScore feel wrong for the same reason unit tests without assertions feel wrong — they don't encode what you care about. LLM-as-judge isn't a substitute for human judgment. It's a scaling technique for judgment you've already made.

Method · Critique Shadowing
Hamel Husain
01
Find THE domain expert.
Singular. A lawyer, a pension consultant, a psychologist. Not a committee. Not you-as-proxy unless you're genuinely qualified.
02
Binary pass/fail + critique.
Not 1–5. One question: did the AI achieve the desired outcome? Critiques explain reasoning and become few-shot examples for the judge.
03
Align the judge iteratively.
Embed critiques as few-shot. Measure precision & recall against the expert — not accuracy. Iterate until agreement exceeds 85%.
04
Error-analyze root causes.
Segment error rates by feature / scenario / persona. Classify failures. Count buckets. Attack the biggest. Repeat, forever.

Framework · Jason Liu

Only 6 RAG Evals.

Three variables — question, context, answer — yield exactly six conditional relationships. Every RAG failure is one of these breaking. No hidden variables.

Tier 2 · Primary
C | Q
Context Relevance
Does retrieval find passages that address the question? If retrieval fails, generation is doomed.
Tier 2 · Primary
A | C
Faithfulness
Does the answer restrict itself to claims supported by retrieved context? The hallucination check.
Tier 2 · Primary
A | Q
Answer Relevance
Does the answer actually address what was asked? The end-to-end UX metric.
Tier 3 · Advanced
C | A
Context Coverage
Does context contain everything needed to support every claim in the answer?
Tier 3 · Advanced
Q | C
Answerability
Given this context, is the question answerable at all? If not — refuse.
Tier 3 · Advanced
Q | A
Self-Containment
Can the question be inferred from the answer alone? Matters for quoted / async contexts.

Same agent.
Different metric.

pass@k
At least one
attempt succeeds.
~100%
Rises with k. Use when one correct solution is enough — coding, brainstorming, research. More shots on goal.
pass^k
All attempts
succeed.
5.6%
Falls with k. Use for customer-facing agents where users expect reliability every time. Consistency is harder.
The same 75% agent looks heroic or broken depending on which metric you chose.
Grader Type 01
Fast · Cheap · Brittle
Code-based.
Methods
  • String match, regex, fuzzy
  • Static analysis (lint, type)
  • Outcome state checks
  • Tool-call verification
  • Transcript metrics
Prefer deterministic graders where possible. They tell you exactly why something failed.
Grader Type 02
Flexible · Scalable · Needs Calibration
Model-based.
Methods
  • Rubric scoring
  • Pairwise comparison
  • Reference-based evaluation
  • Multi-judge consensus (PoLL)
  • Natural-language assertions
Flexible, but non-deterministic. Must be calibrated against human experts — otherwise you're measuring the judge, not the system.
Grader Type 03
Gold Standard · Slow · Expensive
Human.
Methods
  • SME review
  • Crowdsourced judgment
  • Spot-check sampling
  • A/B testing
  • Inter-annotator agreement
Use judiciously, mostly to calibrate the other two. You can't ship an agent evaluated only by humans — but you can't trust one evaluated without them.
