The One Thesis That Ties It All Together

All 11 sources are arguing the same worldview, phrased differently:

Evals are not a metric. Evals are a practice.

The practice is: look at your data, define binary failure modes with a domain expert, and align scalable graders to that expert's judgment. The "metric dashboard" is a downstream artifact. The work is the looking.

Why this framing matters

Once you believe this, the rest snaps into place. Generic metrics (ROUGE, BERTScore, "helpfulness 1–5") feel wrong for the same reason unit tests without assertions feel wrong — they don't encode what you actually care about. LLM-as-judge is a scaling technique for a human judgment you've already made, not a substitute for having made one.

Sources Synthesis across all 11 bibliography entries. Thesis framing most directly from Hamel Husain, "Using LLM-as-a-Judge For Evaluation" (2024) and Anthropic, "Demystifying Evals for AI Agents" (2026).

So… What Do Evals Actually Do?

Strip away the jargon and evals do three jobs:

🔍 Force You to Look at Data

60–80% of LLM project time should go to error analysis. Evals are a trick to make teams do that painful, high-ROI work — categorizing real failures instead of arguing about intuition.

📐 Encode Product Intent

Two engineers reading the same PRD will disagree on edge cases. An eval suite resolves that ambiguity in executable form — it's a spec a machine can check.

🚀 Unlock Fast Iteration

Teams with evals upgrade models in days. Teams without evals spend weeks vibe-checking. The suite becomes the highest-bandwidth channel between product and research.

Why it matters

Every piece of wisdom in the 11 links is downstream of these three jobs. The rest of this primer is just concrete techniques for doing them well.

Sources Hamel Husain, "LLM Evals: Everything You Need to Know" (for the 60–80% error-analysis budget claim). Anthropic, "Demystifying Evals for AI Agents" (spec ambiguity; days-not-weeks model upgrades). Hugo Bowne-Anderson & Stefan Krawczyk, "Escaping POC Purgatory" (evals as product-research interface).

Foundation

The Move Everything Else Builds On: Error Analysis

Andrew Ng has been teaching this for a decade. Every other technique is a tool to scale it.

The Error Analysis Loop

1. Collect→50–100 failed examples
2. Categorize→Tag each by failure type in a spreadsheet
3. Count→Compute % of failures in each bucket
4. Attack→Fix the biggest bucket first
5. Repeat→Forever. This never ends.

💡 Hamel's Reframe

"The real value of an LLM judge is that it tricks people into looking at their data. The judge doesn't create value — the looking does. Potato, potahto."

📊 Data Literacy Matters

Reading traces = EDA. Validating a judge = Model Evaluation. Building test sets = Experimental Design. Classical data science skills are the rate-limiter most LLM teams are missing.

⚠️

Budget warning

Expect 60–80% of serious project time to go to error analysis, not to building evaluators or dashboards. If you're spending less, you're probably building the wrong thing.

Sources Andrew Ng, "Carrying Out Error Analysis" (YouTube, deeplearning.ai). Hamel Husain, "Using LLM-as-a-Judge For Evaluation" and "The Revenge of the Data Scientist" (60–80% budget claim; data-literacy argument). Deep Dive

Critique Shadowing: Hamel's 7-Step Method

This is the most concrete methodology in the entire corpus. Memorize it.

1. Find The Principal Domain Expert
Singular. A lawyer for a legal bot, a pension consultant for an allocator tool, a psychologist for mental-health AI. Not a committee, not you-as-proxy, not "leadership."
2. Build a Diverse Dataset
Use dimensions that match your use case — a starter taxonomy is Features × Scenarios × Personas. Synthetic data works better than you'd think (Hex ships on it). Mix synthetic inputs with real traces. 30+ examples, keep going until new failure modes stop appearing.
3. Binary Pass/Fail + Written Critique
The expert makes one decision: did the AI achieve the desired outcome? Pass or fail. Then writes a critique detailed enough that a new employee could understand the standard. These critiques become few-shot examples for your judge.
4. Fix Obvious Errors First
If review surfaces systemic bugs, stop and fix them before building a judge. The judge should measure a stable system, not a broken one.
5. Build the LLM Judge Iteratively
Prompt with few-shot (input, output, critique, verdict). Send outputs to the expert, track precision and recall (not accuracy — it lies when classes are imbalanced). Iterate the prompt until agreement exceeds 85–90%.
6. Error Analysis on Held-Out Data
Segment error rates across your dimensions. Classify failures by root cause ("Missing User Education", "Auth Issue", "Poor Context Handling"). Fix biggest cluster.
7. Specialized Judges Only If Needed
Don't pre-build citation-checkers, hallucination-detectors, etc. Let error analysis tell you where specialized graders earn their complexity. Often a code assertion is simpler and sufficient.
Source Entire methodology from Hamel Husain, "Using LLM-as-a-Judge For Evaluation: A Complete Guide" (hamel.dev, Oct 2024). Features × Scenarios × Personas taxonomy and the "30+ examples, stop when failure modes stabilize" heuristic come from the same post.

Criteria Drift: Why You Can't Define Success First

From Shankar et al. (Who Validates the Validators?)

"To grade outputs, people need to externalize and define their evaluation criteria; however, the process of grading outputs helps them define that very criteria. We dub this phenomenon criteria drift, and it implies that it is impossible to completely determine evaluation criteria prior to human judging of LLM outputs."

Translation: the rubric emerges from grading, not before it. Any methodology that requires a perfect rubric up-front is doomed. Hamel's critique-shadowing process is designed to exploit this — the expert discovers what they actually care about by grading.

Case Study: Honeycomb Query Assistant

🛠️ The Setup

Hamel worked with Honeycomb's Phillip Carter to evaluate AI-generated Honeycomb queries. Domain expert (Phillip) rated NLQ→query pairs pass/fail with critiques. Judge prompt used his critiques as few-shot examples.

📈 The Result

Three iterations to reach >90% agreement between the LLM judge and Phillip. More interesting: Phillip said grading the LLM's critiques helped him articulate criteria he hadn't realized he held. The judge surfaced the expert's own implicit standards.

🎯

The meta-lesson

Building the judge is valuable even if you never deploy it, because the process forces the expert to make their standards explicit. The judge is a forcing function for institutional clarity.

Sources "Criteria drift" concept from Shreya Shankar, J.D. Zamfirescu-Pereira, Björn Hartmann, Aditya G. Parameswaran, Ian Arawjo, "Who Validates the Validators? Aligning LLM-Assisted Evaluation of LLM Outputs with Human Preferences" (arXiv:2404.12272, 2024). Honeycomb case study from Hamel Husain's "Using LLM-as-a-Judge" post, with domain expert Phillip Carter.
The Trap

Why 1–5 Likert Scales Are a Lie

The single most common beginner mistake: ask an LLM to rate each output on a 1–5 scale across 8 dimensions, average, ship a dashboard. This breaks for specific, documented reasons.

❓ Not Actionable

What does a 3 mean? What separates 3 from 4? Nobody knows and no two raters agree. You can't say "this failed because…" when your output is a number.

📊 Middle Clustering

LLM judges pile up on 6s and 7s on 1–10 scales, 3s on 1–5 scales. You lose discriminative power exactly where you need it most.

📏 Verbosity Bias

Longer answers score higher even when padded with filler. Pass/fail is harder to game; you either did the thing or you didn't.

🔀 Position Bias

In pairwise comparisons, most judges prefer the first option by default — sometimes 50–70% of the time. Randomize or counterbalance to detect this.

🪞 Self-Enhancement Bias

Judges rate outputs from their own model family higher. GPT-4 favors itself ~10%; Claude family ~25%. Mix families if pairwise.

🌫️ Metric Sprawl

"Measure these 8 dimensions" = someone is guessing. Force binary pass/fail first and let the expert articulate what actually matters. Complexity later, not up front.

Hamel's stance

"If your evaluations consist of a bunch of metrics that LLMs score on a 1–5 scale, you're doing it wrong." He means it. Across 30+ implementations he's seen, domain-expert judgments rarely correlate with these scores.

Sources Aishwarya Naresh Reganti, "The 5-Star Lie You Are Doing AI Evals" (Decoding AI). Hamel Husain, "Using LLM-as-a-Judge" (binary-is-better argument). Bias literature from Lianmin Zheng et al., "Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena" (arXiv:2306.05685) — position, verbosity, and self-enhancement bias — as surveyed by Eugene Yan, "Evaluating the Effectiveness of LLM-Evaluators."
The Trap

The Mirage of Generic AI Metrics

BERTScore, ROUGE, BLEU, cosine similarity, "helpfulness", "coherence", "relevance" — generic off-the-shelf metrics almost never correlate with what your domain expert actually cares about.

❌ The Wrong Flow

Framework → dashboard → investigation

Team adopts a framework shipping 15 pre-built judges. Dashboard looks impressive. Metrics don't move when product quality changes. Team loses faith in evals entirely.

✅ The Right Flow

Investigation → failure mode → targeted metric

Error analysis surfaces a pattern. You write a binary check for that specific failure. The metric moves when quality moves because it measures a real thing.

Empirical Correlations (from Eugene Yan's survey)

Human expert ↔ Human expert0.80 – 0.90
LLM judge ↔ Human expert0.30 – 0.60
ROUGE / BERTScore ↔ Human≈ 0.30
🎯

The 70% rule

If you're passing 100% of your evals, your eval isn't stressing your system. A 70% pass rate on meaningful tasks is more valuable than 95% on generic ones. Capability evals should start low and climb; regression evals stay near 100%.

Sources Aishwarya Naresh Reganti, "The Mirage of Generic AI Metrics" (Decoding AI). Correlation numbers from Eugene Yan, "Evaluating the Effectiveness of LLM-Evaluators" (synthesizing SummEval and related benchmarks from Gao et al. arXiv:2304.02554 and Luo et al. arXiv:2303.15621). 70% rule from Hamel Husain, "LLM Evals: Everything You Need to Know" (evals-faq).
Decision Tree

LLM-as-Judge Done Right

Eugene Yan synthesizes two dozen papers into this tree. Internalize the branches.

Q1: Is the task objective or subjective?
• Objective (factuality, toxicity, instruction-following) → direct scoring
• Subjective (tone, persuasiveness, style) → pairwise comparison

Q2: If direct scoring, binary or Likert?
• Binary → classification metrics: precision, recall, Cohen's κ
• Likert → correlation: Spearman's ρ, Kendall's τ (NOT Cohen's κ — over-penalizes ordinal)

Q3: Dev-time evaluator or production guardrail?
• Dev-time (100s of samples, tolerate latency) → biggest model + CoT + few-shot
• Production (low latency, high throughput) → fine-tune small classifier, bootstrap from dev-time judge

Q4: Reference-based or reference-free?
• Reference-based → essentially fuzzy matching; "contains" baselines often beat half of LLM judges
• Reference-free → harder, needs rubric + careful calibration
One empirical surprise worth knowing

A panel of smaller judges (Command-R + GPT-3.5 + Haiku, majority vote) can beat a single GPT-4 judge at 1/7th the cost. This is the "PoLL" approach — useful when latency or budget matter.

Sources Entire decision tree from Eugene Yan, "Evaluating the Effectiveness of LLM-Evaluators (aka LLM-as-Judge)" (eugeneyan.com, Aug 2024) — a synthesis of 24+ papers. PoLL (Panel of LLMs) approach from Pat Verga et al., "Replacing Judges with Juries: Evaluating LLM Generations with a Panel of Diverse Models" (arXiv:2404.18796).
Framework

RAG Evals: There Are Only 6

Jason Liu's framework is elegant because it's exhaustive by construction. A RAG system has three variables: Q (question), C (context), A (answer). Three variables yield exactly six conditional relationships. Every RAG failure is one of these breaking.

Tier 2 — Primary (start here)
C | Q
Context Relevance
Does retrieved context address the question? If retrieval fails, generation is doomed.
A | C
Faithfulness / Groundedness
Does the answer restrict itself to claims supported by context? Hallucination check.
A | Q
Answer Relevance
Does the answer actually address what was asked? End-to-end UX metric.
Tier 3 — Advanced
C | A
Context Coverage
Does context contain everything needed to support every claim in the answer?
Q | C
Answerability
Given this context, is the question answerable at all? If not, refuse.
Q | A
Self-Containment
Can the question be inferred from the answer alone? Matters for async / quoted contexts.

Osmosis context

For our real-time allocator intelligence, A | C (faithfulness) is the most important relationship — hallucinating that a pension fund said something they didn't would be catastrophic. Q | C (answerability) matters almost as much — the right move when an allocator question can't be answered from available transcripts is to say so, not to improvise.

Source Entire framework from Jason Liu, "There Are Only 6 RAG Evals" (jxnl.co, May 2025). Domain-emphasis guidance (medical vs. customer service vs. technical docs) from the same post.

Agent Evals: The Vocabulary

Agents are harder than single-turn systems because they take many steps, call tools, modify external state, and can "succeed" in unexpected ways. Anthropic's piece is the current canonical reference. Adopt this vocabulary — it pays compounding returns in team conversations.

TermMeaning
TaskOne test case with inputs and success criteria (a.k.a. "problem")
TrialOne stochastic run of a task. Multiple trials average out variance.
GraderLogic that scores some aspect of the agent's performance. Tasks can have multiple graders.
Transcript / TraceFull record of the trial — outputs, tool calls, reasoning, intermediate results
OutcomeThe final state in the environment — not what the agent said, what actually happened (e.g. reservation exists in DB)
Eval harnessInfrastructure that runs tasks, records steps, grades outputs, aggregates results
Agent harness / scaffoldThe system that makes a model act as an agent. When you evaluate "an agent," you're evaluating model + scaffold together.
Evaluation suiteA collection of tasks testing specific capabilities (e.g., "customer support evals": refunds, cancellations, escalations)
Key insight: outcome ≠ transcript

A flight-booking agent that says "Your flight has been booked!" might still have failed to actually call the reservation tool. Always grade the outcome (state in the environment), not just what the agent claimed it did.

Source Entire vocabulary from Anthropic, "Demystifying Evals for AI Agents" (anthropic.com/engineering, Jan 2026), by Mikaela Grace, Jeremy Hadfield, Rodrigo Olivares, and Jiri De Jonghe. Flight-booking example from the same post, referencing τ²-Bench (Yao et al., arXiv:2506.07982).
Framework

Three Types of Graders — Pick the Right One for Each Check

Great eval design combines all three. Prefer deterministic where possible, LLM where necessary, human judiciously.

⚙️ Code-Based

Methods: regex, string match, static analysis, outcome state checks, tool-call verification, transcript metrics

Strengths: Fast, cheap, objective, reproducible, easy to debug

Weaknesses: Brittle to valid variations; no nuance; can't judge subjective tasks

🤖 Model-Based

Methods: rubric scoring, natural-language assertions, pairwise comparison, reference-based, multi-judge consensus

Strengths: Flexible, scalable, handles open-ended output, captures nuance

Weaknesses: Non-deterministic, expensive, needs calibration against humans

🧑‍⚖️ Human

Methods: SME review, crowdsourced judgment, spot-check sampling, A/B testing, inter-annotator agreement

Strengths: Gold standard; matches expert user judgment; calibrates LLM judges

Weaknesses: Expensive, slow, doesn't scale; access to experts is a bottleneck

💡

Capability vs. regression evals

Capability evals should start low — they give the team a hill to climb. Regression evals should stay near 100% — they catch backsliding. As capability evals saturate, they graduate into the regression suite.

Source Grader taxonomy and capability/regression distinction from Anthropic, "Demystifying Evals for AI Agents." Underlying benchmarks referenced: SWE-bench Verified, Terminal-Bench, τ-Bench (Yao et al., arXiv:2406.12045).
Framework

pass@k vs. pass^k — Picking the Right Reliability Metric

Agents are stochastic. A task that passes on one run may fail on the next. Two metrics capture this — use whichever matches your product requirement.

🎯 pass@k

"At least one correct solution in k attempts."

As k grows, pass@k rises — more shots on goal means higher odds of at least one success. Use when one correct solution is enough (coding, brainstorming, research).

🛡️ pass^k

"All k trials succeed."

As k grows, pass^k falls — demanding consistency is a higher bar. Use when users expect reliable behavior every time (customer-facing agents, payment flows, medical).

How fast they diverge

Per-trial success rate=75%
pass@1 = pass^1=75%
pass@10 (at least 1 success)=~100%
pass^10 (all 10 succeed)=~5.6%
Why this matters

At k=1 they tell the same story. By k=10 they tell opposite stories. A 75% pass@1 agent looks amazing (pass@10 ≈ 100%) or terrible (pass^10 ≈ 5.6%) depending on what you care about. Pick the metric that matches your product's actual reliability requirement.

Sources Framing from Anthropic, "Demystifying Evals for AI Agents." pass@k introduced in Chen et al., "Evaluating Large Language Models Trained on Code" (arXiv:2107.03374, Codex paper, 2021). pass^k introduced in Yao et al., "τ-Bench: A Benchmark for Tool-Agent-User Interaction in Real-World Domains" (arXiv:2406.12045, 2024).
Counterpoint

LLM Judges Aren't the Shortcut You Think

Doug Turnbull comes at this from search / relevance engineering, which is a useful counterweight to Hamel and Shreya's product-AI framing. His caution is worth taking seriously.

The core warning

LLM judges approximate one kind of human evaluation: topical relevance — "is this document about what the query asked about?" LLMs are great at this because they live in a world of facts. But relevance in a real product is driven by user "lizard brains" — intent, personalization, recency, authority, click-through, conversion. No LLM judge captures those.

🏆 Popularity Bias

LLM judges favor "canonical" or well-known entities. Your niche-but-correct result looks wrong to the judge.

📏 Verbosity Bias

Long well-formed prose beats sparse but correct data. For structured-data products this is actively wrong.

🎨 Style Bias

Judges inherit training-data aesthetics. "Feels like a good answer" doesn't always match domain signals of quality.

🔁

Turnbull's alternative framing

Instead of "LLM as judge," consider LLM features in a smaller scoring model. Use the LLM's embeddings, ranked outputs, or structured judgments as features in a learned ranker or cross-encoder. You keep the LLM's knowledge, discard its opinions, and avoid many calibration headaches.

Sources Doug Turnbull, "LLM Judges Aren't the Shortcut You Think" (softwaredoug.com, Nov 2025). Self-enhancement and verbosity bias evidence from Lianmin Zheng et al., "Judging LLM-as-a-Judge with MT-Bench and Chatbot Arena" (arXiv:2306.05685). The YouTube talk linked in the original bibliography may be a different speaker (possibly Shreya Shankar) covering adjacent ground — treatment here is based on the blog post of the same title.

Cheat Sheet

A 4-Week Plan to Go From Zero to Trustworthy Evals

If you put this into practice tomorrow, here's the concrete path. Nothing fancy — just the moves that compound.

Week 1 — Set up the looking
Identify your Principal Domain Expert. Pull 30–50 real production traces. Build a simple web viewer so the expert sees each trace with all context on one screen. Expert marks pass/fail + writes a critique on each. Do not build anything else yet.
Week 2 — Categorize and fix
Review critiques as a team. Cluster failures by root cause. Fix the biggest 2–3 bugs directly (prompt tweaks, retrieval parameter changes, tool-description edits). Track what changed and why.
Week 3 — Build the judge
Write an LLM judge prompt with 5–10 critique examples embedded. Run it on the same 30–50 traces. Compare judge verdicts to expert labels. Iterate until agreement >85% — measured by precision & recall, not accuracy.
Week 4 — Close the loop
Run judge on fresh held-out traces. Segment error rates by feature / scenario / persona. Identify next 2–3 improvements to ship. Wire judge into CI so every PR runs it.
Ongoing
Every bug report becomes an eval case. Every model release triggers a run. Every new feature ships with an eval defining what "done" means. Sample and read 10 transcripts per week.
🚫

The one warning every source agrees on

Don't optimize for high pass rates. 100% means your eval isn't working. Add harder tasks. The point is to have a hill to climb.
